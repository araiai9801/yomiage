WordもEdgeも要らない──どんなアプリのテキストも一瞬で音声化するホットキーツールを自作した

なぜ作ったのか

緑内障を患っており、両目とも視力が相当悪い。近眼とは違い、眼鏡をかけても視力はほとんど変わらない。背景黒画面に白文字は比較的読めるが、白画面に黒文字はとても読みづらい。200％拡大画面スクロールしながら追う日常が続いている。

Word や Edge には優秀な読み上げ機能がある。とても助かっている。しかし世の中のソフトすべてがそうではない。メモ帳、PDF ビューア、SNS、社内ツール——読み上げに対応していないアプリはまだ多く、そのたびにコピペしてWordに貼り付けて読み上げさせるか、拡大しながら目を酷使するしかなかった。「どんなアプリでも、選択してキーを押すだけで読んでくれるものがあれば」。それが出発点だった。

完成まで数週間かかった理由

Claude Code に最初の指示を出したのは4月の中頃。ClaudeCodeのバイブコーディングくぉ試そうと思い手始めに自分の困りごとからスタートした。「Windows で選択テキストをホットキーで読み上げるアプリを作ってほしい」。Claude はすぐに動いた。Microsoft の edge-tts（Nanami Neural という高品質な日本語音声）を使い、グローバルホットキーで動く Python アプリをあっという間に生成した。

ところがそこからが長かった。

最初の難関は「一時停止」の仕様だ。当初は Esc キーで即停止するだけだったが、使っているうちに「長い文章の途中で止めたい」という欲求が出てきた。Tab キーで一時停止・再開できないか、と仕様を追加した。Claude はすぐ実装してくれたが、試してみると Tab を押しても止まらない。

何度修正を依頼しても変わらない。「チャンクが短くなっていない」「キーが効かない」と操作の感想を伝え続けること数十回。こりゃダメだ、こんなものかとあきらめかけていた。

ある日、「もしかして、Claude が編集しているファイルと、私が起動しているファイルは同じですか？」と聞いてみた。

Claude がレジストリを調べると——別ファイルだった。

私はデスクトップの右クリックメニューから yomiage\yomiage.py を起動していた。一方、Claude が修正し続けていたのは別の screen_reader.py というファイル。何セッションもかけて積み上げた修正が、まるごと無関係なファイルへの変更だったのだ。気づいたのは Claude で、私ではない。ただ「違うかもしれない」と疑問を口にしただけだ。直したはずのファイルが直っていないのは、実は別ファイルだったというのはよくある話だ。これがWordやExcelならすぐに気が付いたはずだ。プログラムコードの場合は何かバグなのか？と疑ってしまい、修正箇所の間違いに気付かないことがある。ましてバイブコーディングなので、コードを見るわわけでもないのでなおさらだ。これは落とし穴ですね。ともあれ、違うファイルだとわかったときは、アホさ加減に笑うしかなかった。当然違うファイルを見ているなどと思うはずもない。思い込みは間違いの元。Claudeは私が操作してるアプリを知らず私はClaudeがいじっているCodeを見ていない。バイブコーディングのあるあるなのかも。まあこういうことは徐々に慣れていくしかない。

仕様の変遷と技術的な躓き

正しいファイルを修正対象に定めてからも、試行錯誤は続いた。

チャンク分割の調整。長文を文節ごとに区切って順に読み上げる仕組みなのだが、1チャンクが長すぎて Tab を押しても「最後まで読んでから止まる」まで体感的に長く感じる。チャンクの最大文字数を 120 → 50 に縮めると、今度はチャンクとチャンクの間に無音の「間」が生じて不自然な読み上げになった。

Claude がコードを調べると、プリフェッチ（次のチャンクを再生中に並行生成する仕組み）が実は機能していなかった。先読みして生成したはずの MP3 ファイルを、次のイテレーションでまた一から生成し直すバグがあり、プリフェッチが丸ごと無駄になっていた。修正後、チャンク間のギャップはほぼ消えた。私がしたのは「チャンクの間が空きすぎる」と伝えただけだ。

Tab キーの動作仕様も変わった。最初は音声を途中でブツッと止める実装だったが、実際に聴いてみると不自然だった。「今の文節が終わってから止まってほしい」と伝えると、Claude が方式を切り替えた。私はコードを見ていない。聴いた感想を言っただけだ。

KEYEVENTF_EXTENDEDKEY の罠。「カーソル位置から文末まで読む」機能が動かない時期も長かった。私には何が起きているか分からない。「動かない」と伝え続けた。Claude がログを追い、Windows の SendInput API における拡張キーフラグの欠落を突き止め、修正した。私の役割はひたすら「これは違う」と言い続けることだった。

使ってみて

完成したアプリは、メモ帳でも PDF でも Line でも、テキストを選んで Ctrl+Alt+R を押すだけで読み上げが始まる。Tab で文節の区切りに一時停止/再開、Esc で即停止。画面下部には今読んでいる文節がオーバーレイ拡大表示される。目が弱い自分には、これが特にありがたい。

今後の改良点

長い文の場合、何度も選択・読み上げを繰り返すのは面倒なのだが、Ctrl+Alt+E でカーソル位置から最後の行まで読み上げてくれる。これは結構愛用している。但し、この機能はすべてのアプリが対応しているわけではない点が残念だが、、、もう一つ改良したいのは、PowerPointの場合、文字列がTextboX内にあるため、その中でしか選択できない点である。最近のAI生成でのパワポ資料は、テキストを1行単位で細切れにtextbox化するので、選択ー読み上げを何度も繰り返さなければならない。まとめて選択する方法があればよいのだが。
