# WordもEdgeも要らない──どんなアプリのテキストも一瞬で音声化するホットキーツールを自作した

## なぜ作ったのか

緑内障を患っており、両目の視力は 0.15 と 0.3 だ。近眼とは違い、眼鏡をかけても視力はほとんど変わらない。画面は常に 200% 前後に拡大して使い、それでも読みにくい文字は目を細めてスクロールしながら追う日常が続いている。

Word や Edge には優秀な読み上げ機能がある。とても助かっている。しかし世の中のソフトすべてがそうではない。メモ帳、PDF ビューア、社内ツール——読み上げに対応していないアプリはまだ多く、そのたびに目を酷使するしかなかった。「どんなアプリでも、選択してキーを押すだけで読んでくれるものがあれば」。それが出発点だった。

## 完成まで数日かかった理由

Claude Code に最初の指示を出したのは数日前。「Windows で選択テキストをホットキーで読み上げるアプリを作ってほしい」。Claude はすぐに動いた。Microsoft の edge-tts（Nanami Neural という高品質な日本語音声）を使い、グローバルホットキーで動く Python アプリをあっという間に生成した。

ところがそこからが長かった。

最初の難関は「一時停止」の仕様だ。当初は Esc キーで即停止するだけだったが、使っているうちに「長い文章の途中で止めたい」という需要が出てきた。Tab キーで一時停止・再開できないか、と仕様を追加した。Claude はすぐ実装してくれたが、試してみると **Tab を押しても止まらない**。

何度修正を依頼しても変わらない。「チャンクが短くなっていない」「キーが効かない」とフィードバックを続けること数セッション。ある日ふと気づいた。「あれ、Claude が編集しているファイルって、自分が起動しているファイルと同じ？」

確認してみると——**別ファイルだった**。

自分はデスクトップの右クリックメニューから `yomiage\yomiage.py` を起動していた。一方、Claude が修正し続けていたのは別の `screen_reader.py` というファイル。レジストリを調べて初めて判明した。何セッションもかけて積み上げた修正が、まるごと無関係なファイルへの変更だったのだ。原因が分かった瞬間、笑うしかなかった。

## 仕様の変遷と技術的な躓き

正しいファイルを修正対象に定めてからも、試行錯誤は続いた。

**チャンク分割の調整**。長文を文節ごとに区切って順に読み上げる仕組みなのだが、1チャンクが長すぎて Tab を押しても「最後まで読んでから止まる」まで体感的に長く感じる。チャンクの最大文字数を 120 → 50 に縮めると、今度はチャンクとチャンクの間に無音の「間」が生じて不自然な読み上げになった。

原因を調べると、プリフェッチ（次のチャンクを再生中に並行生成する仕組み）が実は機能していなかった。コードをよく見ると、先読みして生成したはずの MP3 ファイルを、次のイテレーションでまた一から生成し直していた。つまりプリフェッチが無駄になり、毎回ネットワーク待ちが発生していた。先読みスレッドの結果をループをまたいで引き継ぐよう修正したところ、チャンク間のギャップがほぼ消えた。

**Tab キーの動作仕様も変わった**。最初は「MCI の pause コマンドで音声を途中停止」という実装だったが、音声が唐突に途切れるため不自然だった。「現在の文節を最後まで読み終えてから止まる」という仕様に変更し、チャンクの切れ目でフラグをチェックする方式に切り替えた。これで自然な聴き心地になった。

**KEYEVENTF_EXTENDEDKEY の罠**。「カーソル位置から文末まで選択して読む」機能でも詰まった。Ctrl+Shift+End を SendInput で送っているのに選択されない。ログを追うと、End キーは Windows の「拡張キー」であり、SendInput API では `KEYEVENTF_EXTENDEDKEY` フラグを立てないと Shift が無視されてしまうと判明。フラグを 1 つ足すだけで解決した。地味だが、知らなければ永遠に詰まる落とし穴だった。

## 使ってみて

完成したアプリは、メモ帳でも PDF でも Slack でも、テキストを選んで Ctrl+Alt+R を押すだけで読み上げが始まる。Tab で文節の区切りに一時停止、Esc で即停止。画面下部には今読んでいる文節がオーバーレイ表示される。目が弱い自分には、これが特にありがたい。

数日かけて完成したこのツール、コードは GitHub で無料公開中だ。Windows ユーザーの方はぜひ。
